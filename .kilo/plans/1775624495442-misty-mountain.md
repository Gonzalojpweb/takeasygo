# Plan: Implementar Funcionalidad de Pausar/Reactivar Tenants

## Contexto
Actualmente, los tenants se eliminan definitivamente cuando ya no se necesitan (e.g., demos expirados). Queremos implementar una funcionalidad para **pausar** tenants en lugar de eliminarlos, preservando toda la información para posibles reactivaciones futuras.

## Objetivos
- Preservar la información de tenants (menús, configuración, pedidos históricos) sin "regalarla"
- Permitir reactivaciones rápidas sin recargar datos
- Diferenciar entre pausa temporal y eliminación definitiva

## Análisis Actual
- El modelo `Tenant` tiene `isActive: boolean` usado en ~90 lugares para filtrar tenants activos
- La eliminación actual es definitiva (DELETE)
- No hay distinción entre "inactivo temporal" vs "eliminado"

## Propuesta de Implementación

### 1. Modificar Modelo de Datos
**Cambios en `models/Tenant.ts`:**
- Agregar campo `status: 'active' | 'paused' | 'deleted'`
- Mantener `isActive` como computed: `active || paused` (para compatibilidad)
- Agregar `pausedAt: Date | null` y `pausedReason: string`

**Migración:**
- Tenants existentes: `status = 'active'`
- Mantener `isActive = true` por compatibilidad

### 2. Lógica de Estado
**Estados del tenant:**
- `active`: Funciona normalmente
- `paused`: Visible para admin, pero no puede recibir nuevos pedidos, no aparece en explore
- `deleted`: Eliminado definitivamente

**Comportamiento por estado:**
- `paused`: 
  - No puede crear pedidos nuevos
  - Menús no visibles en explore
  - Admin puede acceder para gestión
  - Pedidos existentes siguen visibles
  - No puede editar configuración

### 3. APIs Nuevas
**Superadmin:**
- `POST /api/superadmin/tenants/[tenantId]/pause` - Pausar tenant
- `POST /api/superadmin/tenants/[tenantId]/resume` - Reactivar tenant

**Request body para pause:**
```json
{
  "reason": "Demo expirado - esperando decisión del cliente"
}
```

### 4. Cambios en Queries Existentes
**Actualizar filtros:**
- Donde dice `isActive: true` → `status: { $in: ['active', 'paused'] }`
- Explore endpoints: solo `status: 'active'`

**Endpoints que cambian:**
- Creación de pedidos: verificar `status === 'active'`
- Listados de admin: mostrar todos excepto `deleted`
- Explore: solo `active`

### 5. UI en Superadmin
**Vista de tenants:**
- Columna "Estado" con badges: Activo/Pausado/Eliminado
- Botones: Pausar/Reactivar/Eliminar
- Filtros por estado

**Modal de pausa:**
- Razón obligatoria
- Confirmación "El tenant seguirá siendo visible para admin pero no podrá recibir pedidos"

### 6. UI para Admin de Tenant Pausado
**Página especial de acceso denegado:**
- Cuando un admin entra a un tenant pausado, NO ve el panel normal
- En su lugar, ve una página dedicada con:
  - Mensaje: "Tu cuenta está temporalmente pausada. Para más información sobre el acceso, comunicate con el equipo de soporte."
  - Botones con WhatsApp:
    - Soporte: +5491160019734
    - Comercial: +5491138795976
- No tiene acceso a ninguna funcionalidad del panel
- Diseño limpio y profesional

**Confirmado:** Sin notificaciones automáticas al pausar

### 7. Consideraciones de Seguridad
- Solo superadmin puede pausar/reactivar
- Logs de auditoría para todos los cambios de estado
- Tenants pausados no pueden ser accedidos por users normales

### 8. Migración de Datos
- Script para actualizar tenants existentes
- Backup antes de migrar

## Tradeoffs a Evaluar

**Pregunta 1:** ¿Los tenants pausados deben aparecer en explore?
- Opción A: No aparecen (privacidad)
- Opción B: Aparecen pero con banner "Temporalmente cerrado"

**Confirmado:** Los tenants pausados NO aparecen en explore (opción privada)

**Confirmado:** Nunca se pausa un tenant con pedidos en progreso. Los pedidos existentes se completan normalmente antes de pausar.

**Confirmado:** Sin límite de tiempo para tenants pausados (pueden estar pausados indefinidamente)

## Riesgos
- Queries rotas si no se actualizan todos los `isActive: true`
- Tenants pausados podrían seguir recibiendo pedidos por webhooks
- Complejidad adicional en UI de admin

## Beneficios
- Recuperación rápida de clientes perdidos
- Mejor experiencia para demos
- Reducción de tiempo de setup para reactivaciones
